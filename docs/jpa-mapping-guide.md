# LTSS JPA mapping guide

## 1. Mapping policy

Flyway owns the schema. Hibernate must run with `spring.jpa.hibernate.ddl-auto=validate`; `create`, `create-drop`, and `update` are prohibited. Entities are added only when a feature uses them, but every mapping must match V1 exactly.

Use feature-first packages, constructor injection, repositories behind services, request/response DTOs at the REST boundary, and explicit aggregate transactions. Never expose a JPA entity directly from a controller.

## 2. Scalar types

| MySQL type | Java/JPA mapping | Rule |
|---|---|---|
| `BIGINT UNSIGNED` identifiers | `Long` | Use `@Id` and `GenerationType.IDENTITY` only for `AUTO_INCREMENT`. The application must reject/alert before values exceed `Long.MAX_VALUE`, although MySQL's unsigned range is larger. |
| Other `BIGINT UNSIGNED` values | `Long` | Validate nonnegative values before persistence. `file_size_bytes` is an example. |
| `INT/SMALLINT/TINYINT UNSIGNED` | `Integer`, `Short`, or `Integer` as appropriate | Java has no unsigned JPA primitive; use wrappers and bean/domain validation. Map optimistic versions to `Integer`. |
| `BOOLEAN` | `Boolean` | Columns are non-null unless the DDL says otherwise. Do not use nullable wrappers to model a third state that the schema does not have. |
| `DECIMAL(p,s)` | `BigDecimal` | Always declare matching `precision` and `scale`; never use `float`/`double` for coordinates, money, distance, duration, scores, or percentages. |
| `CHAR`/`VARCHAR`/`TEXT` | `String` | Match length for bounded columns. Use `@Lob` only when required by the Hibernate/MySQL mapping; validate trimmed/nonblank semantics in DTO/domain code. |
| `DATE` | `LocalDate` | No timezone conversion. |
| `JSON` | `JsonNode` or a dedicated value object | Use Hibernate 6 `@JdbcTypeCode(SqlTypes.JSON)` and `columnDefinition = "json"`, or one reviewed converter. Do not put entity relationships in JSON. |

Representative decimal declarations:

```java
@Column(name = "entrance_fee", precision = 12, scale = 2, nullable = false)
private BigDecimal entranceFee;

@Column(name = "latitude", precision = 10, scale = 7)
private BigDecimal latitude;

@Column(name = "score_percent", precision = 5, scale = 2, nullable = false)
private BigDecimal scorePercent;
```

## 3. UTC `DATETIME(6)`

MySQL `DATETIME` has no timezone. LTSS treats every `DATETIME(6)` value as UTC.

- Prefer `Instant` for service/domain timestamps and configure Hibernate/JDBC timezone as UTC.
- If a driver/dialect mapping uses `LocalDateTime`, it represents UTC only; convert to/from `Instant` at the boundary and never use the host default zone.
- Preserve microsecond precision (`TIMESTAMP(6)`/`DATETIME(6)` mapping) and compare timestamps using an application `Clock` in tests.
- Database-default audit/creation columns may be mapped `insertable = false`; lifecycle fields such as `published_at`, `resolved_at`, and `submitted_at` are set explicitly by the state-transition service.
- Frontend/API serialization uses ISO-8601 UTC instants; Asia/Ho_Chi_Minh conversion is presentation-only.

Do not mix `OffsetDateTime`, local server time, and database session time inside one aggregate.

## 4. Generated values and optimistic locking

Every `id BIGINT UNSIGNED AUTO_INCREMENT` uses:

```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

Map `version INT UNSIGNED` as:

```java
@Version
@Column(nullable = false)
private Integer version;
```

The versioned tables are `users`, `places`, `businesses`, `events`, `articles`, `business_posts`, `promotions`, `tours`, `quizzes`, `reviews`, and `review_replies`. Convert optimistic-lock failures to a safe HTTP conflict response; do not silently overwrite newer state.

`media_assets.storage_key_hash` is generated by MySQL:

```java
@Column(name = "storage_key_hash", insertable = false, updatable = false,
        columnDefinition = "BINARY(32)")
private byte[] storageKeyHash;
```

The service writes the canonical full `storageKey`, looks up by the 32-byte SHA-256 value, and verifies the returned full key. Do not add a second Java-maintained hash column.

## 5. Composite identifiers

The following 14 tables require an `@Embeddable` ID and `@EmbeddedId`:

1. `role_permissions(role_id, permission_id)`
2. `role_inheritances(role_id, inherited_role_id)`
3. `user_roles(user_id, role_id)`
4. `business_post_tags(business_post_id, tag_id)`
5. `favorites(user_id, place_id)`
6. `quiz_badges(quiz_id, badge_id)`
7. `place_media(place_id, media_asset_id)`
8. `event_media(event_id, media_asset_id)`
9. `article_media(article_id, media_asset_id)`
10. `business_post_media(business_post_id, media_asset_id)`
11. `promotion_media(promotion_id, media_asset_id)`
12. `tour_media(tour_id, media_asset_id)`
13. `review_media(review_id, media_asset_id)`
14. `quiz_media(quiz_id, media_asset_id)`

ID classes must be `Serializable`, immutable from the domain perspective, have a no-argument constructor for JPA, and implement equality/hash code only from the key values. A typical association mapping is:

```java
@EmbeddedId
private RolePermissionId id;

@MapsId("roleId")
@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "role_id", nullable = false)
private Role role;
```

Use an explicit association entity whenever the table has attributes such as `granted_at`, `is_active`, `display_order`, or `is_primary`; do not hide it behind a plain `@ManyToMany`.

`relic_details.place_id` is a shared single-column PK/FK and uses `@MapsId`, not `@EmbeddedId`. `tour_items` has a surrogate `id` and is also an explicit association entity.

## 6. Enums and state

Statuses and codes remain `VARCHAR + CHECK`, so Java enums use `@Enumerated(EnumType.STRING)` or a reviewed converter. Enum constant names must match database values exactly. Do not expose public setters for status, owner, moderator, published timestamp, score, badge, or audit fields; state-transition methods/services own them.

Database checks enforce only local row invariants. Services still validate allowed state transitions, permissions, ownership, current version, required timestamps, and cross-row counts in one transaction.

## 7. Polymorphic targets

`reviews`, `engagement_events`, and `moderation_records` contain several nullable, real FKs. Map them as explicit lazy `@ManyToOne` associations. Do not use Hibernate `@Any`, an unvalidated `entityType/entityId` pair, or a JSON target.

Central domain validators must count non-null target IDs and require exactly one before persistence. Each feature exposes target-specific factory methods so controllers cannot construct an invalid multi-target entity. The same transaction performs target existence/ownership checks and writes associated history.

## 8. JSON fields

JSON appears only in:

- `audit_logs.old_values`
- `audit_logs.new_values`
- `engagement_events.metadata`

Use `JsonNode` for immutable snapshots or a narrowly typed value object for known metadata. Validate size and allowed keys before persistence. Redact passwords, password hashes, authorization headers, access/refresh tokens, OTP/reset tokens, cookies, and secrets before an audit snapshot is created.

## 9. Cascades and deletion

JPA cascade is an object-graph persistence choice and does not automatically mirror database `ON DELETE`. Apply these rules:

- Composition/association rows backed by `ON DELETE CASCADE` may use `orphanRemoval`/remove cascade only when deletion is an approved aggregate operation. Examples include tour items, question/answer composition, review replies, and link entities.
- `ON DELETE RESTRICT` relationships never use blanket `CascadeType.REMOVE` or `CascadeType.ALL` from the parent. Archive/soft-delete the aggregate through its service.
- `ON DELETE SET NULL` history/actor relationships remain optional and do not cascade persist/remove from history to user/content.
- Audit, engagement, password history, user badge history, terminal quiz attempts/snapshots, and terminal moderation records are not mutable through general repositories or REST endpoints.
- Default associations to `FetchType.LAZY`; queries/DTO projections load only the data required by a use case.

## 10. REST boundary

Entities are persistence models, not API contracts. Controllers accept validated request DTOs and return response DTOs/envelopes. This avoids lazy-loading leaks, recursion, accidental disclosure of `password_hash`/token hashes/audit internals, and client control of server-owned fields.

Mapping from entity to response occurs inside the feature layer while the required data is loaded. Authorization and ownership checks happen before mapping. File streams and other explicitly special endpoints may bypass the normal response envelope, but still never serialize entities directly.
